# 开发内容梳理

## 1. 目标

本文档将 MVP 方案拆解为可执行开发内容，用于：

- 明确开发范围
- 安排实施顺序
- 识别模块依赖
- 作为开发和联调清单

默认目标版本：**MVP 第一版**

## 2. 开发阶段总览

建议分为 6 个阶段推进：

1. 项目初始化
2. 本地项目管理
3. 配置管理
4. 执行链路打通
5. 日志与状态可视化
6. AI 诊断与收尾优化

## 3. 阶段拆解

## 3.1 阶段一：项目初始化

目标：完成可运行的桌面前端工程骨架。

### 开发内容

- 初始化 `Tauri + Vue3 + TypeScript + Vite`
- 引入 UI 组件库
- 建立基础目录结构
- 配置路由或工作区布局
- 配置状态管理
- 配置代码规范
- 配置环境变量方案

### 输出结果

- 项目可本地启动
- 桌面窗口可正常打开
- 基础页面框架可展示

### 验收标准

- `npm/pnpm run dev` 可启动前端
- `tauri dev` 可启动桌面壳
- 代码结构清晰，基础依赖齐全

## 3.2 阶段二：本地项目管理

目标：完成本地前端项目导入、识别与存储。

### 开发内容

- 接入本地目录选择能力
- 读取目标目录下 `package.json`
- 解析项目名称、脚本、依赖信息
- 识别项目类型
- 识别包管理器
- 推断默认打包命令
- 推断默认输出目录
- 保存项目记录到本地存储
- 渲染项目列表

### 输出结果

- 用户可导入项目
- 导入后能看到基础信息
- 关闭重开后项目记录仍存在

### 验收标准

- 可成功导入 Vite/Vue/React 项目
- 项目列表可展示名称、路径、类型
- 导入异常时有明确错误提示

## 3.3 阶段三：配置管理

目标：完成项目配置和部署环境配置的录入与保存。

### 开发内容

- 项目默认配置表单
- 环境配置表单
- 支持 `dev` / `test` / `prod`
- 配置字段校验
- 本地持久化保存
- 编辑与回填
- 删除配置

### 配置项范围

- 打包命令
- 输出目录
- 前置校验开关
- 前置校验命令
- SSH 主机
- SSH 端口
- 用户名
- 认证方式
- 远端目录
- 上传策略
- 后置命令

### 输出结果

- 用户可对每个项目维护默认配置
- 用户可为项目维护多个环境配置

### 验收标准

- 配置保存后可回显
- 必填项缺失时不可提交
- 环境切换后表单内容正确切换

## 3.4 阶段四：执行链路打通

目标：完成前端到 OpenClaw 网关的任务下发与控制。

### 开发内容

- 封装 WebSocket 客户端
- 建立连接状态管理
- 定义任务请求数据结构
- 定义状态回传处理逻辑
- 实现三类任务入口
- 实现停止任务入口
- 实现任务去重或并发限制

### 任务范围

- 仅打包
- 仅部署
- 打包并部署

### 输出结果

- 前端可下发任务
- 可接收任务生命周期状态

### 验收标准

- 点击执行后能成功发送任务
- 网关返回状态后前端状态同步更新
- 连接断开时有明确提示

## 3.5 阶段五：日志与状态可视化

目标：让用户清楚看到任务过程、进度和失败点。

### 开发内容

- 实时日志面板
- 日志分级渲染
- 当前阶段展示
- 任务状态展示
- 成功/失败结果展示
- 自动滚动到底部
- 错误日志高亮
- 历史任务基础记录

### 输出结果

- 用户可看到完整任务过程
- 失败时可快速定位阶段和错误

### 验收标准

- 日志可实时追加显示
- `info/warn/error/success` 样式可区分
- 任务完成后状态和时间信息完整

## 3.6 阶段六：AI 诊断与收尾优化

目标：提高失败排查效率，补齐 MVP 体验。

### 开发内容

- 失败日志收集
- AI 诊断请求入口
- 诊断结果展示面板
- 常见错误摘要展示
- 项目排序优化
- 表单交互优化
- 加载态和空态优化

### 输出结果

- 任务失败后可展示诊断建议
- 页面交互更完整

### 验收标准

- 失败任务可触发 AI 诊断
- 诊断内容包含原因和建议
- 无数据、连接失败、执行中等状态都有明确反馈

## 4. 模块拆分

## 4.1 视图层

负责页面渲染和交互。

### 建议模块

- 项目列表视图
- 配置工作区视图
- 日志工作区视图
- 全局连接状态组件
- 全局消息提示组件

## 4.2 状态层

负责管理全局状态和运行态状态。

### 建议状态域

- 项目状态
- 环境配置状态
- 当前选中项目状态
- WebSocket 连接状态
- 当前任务状态
- 当前任务日志状态
- AI 诊断状态

## 4.3 服务层

负责非 UI 逻辑。

### 建议模块

- 项目识别服务
- 本地存储服务
- WebSocket 通信服务
- 任务编排服务
- 日志转换服务
- AI 诊断服务

## 4.4 类型层

负责统一约束前后端消息和本地数据结构。

### 建议类型

- 项目模型
- 环境模型
- 执行模型
- 任务模型
- 日志模型
- AI 诊断模型
- WebSocket 消息模型

## 5. 开发优先级

## 5.1 P0

必须先完成，否则不能形成 MVP 闭环。

- 项目初始化
- 项目导入
- 项目识别
- 配置保存
- WebSocket 连接
- 打包任务下发
- 部署任务下发
- 实时日志回显
- 任务状态展示

## 5.2 P1

建议在 MVP 内补齐，提升可用性。

- 多环境切换
- 停止任务
- AI 失败诊断
- 历史任务记录
- 表单校验完善

## 5.3 P2

可以在 MVP 后继续扩展。

- 配置导入导出
- 模板化部署动作
- 多服务器部署
- 批量执行

## 6. 依赖关系

建议按以下依赖顺序开发：

1. 项目初始化
2. 本地存储与类型定义
3. 项目导入与识别
4. 配置表单与保存
5. WebSocket 客户端
6. 任务下发与状态流
7. 日志渲染
8. AI 诊断
9. 体验优化

关键依赖说明：

- 没有类型定义和本地存储，配置管理容易返工
- 没有 WebSocket 协议约束，日志与状态层会混乱
- 没有任务状态流，日志页无法正确工作

## 8. 联调重点

开发阶段要重点验证以下问题：

- WebSocket 连接是否稳定
- 日志流是否存在乱序
- 任务状态是否与日志阶段一致
- 失败任务是否能稳定产出错误摘要
- 项目路径、输出目录、远端路径是否存在边界问题
- 用户重复点击执行时是否会触发重复任务

## 9. 风险点

需要尽早关注以下风险：

- OpenClaw 网关协议不稳定，导致前端反复改消息结构
- 本地敏感信息存储方式不合理
- 项目识别规则过弱，导致自动填充不准确
- 日志量过大时页面渲染性能下降
- 任务中断与 WebSocket 断开后的状态恢复不一致

## 10. 建议的第一周产出

如果按最小闭环推进，第一周建议目标为：

- 工程初始化完成
- 页面基础框架完成
- 本地项目导入完成
- `package.json` 解析完成
- 项目列表展示完成
- 本地存储打通

## 11. 建议的第二周产出

- 配置表单完成
- 环境配置持久化完成
- WebSocket 客户端完成
- 打包任务链路打通
- 日志面板基础版完成

## 12. 建议的第三周产出

- 部署链路打通
- 打包并部署链路完成
- 失败处理完善
- AI 诊断接入
- MVP 联调与修正

## 13. 结论

当前阶段最重要的不是继续扩展需求，而是把开发内容压成一个明确闭环：

1. 导入项目
2. 保存配置
3. 下发任务
4. 查看日志
5. 处理失败

只要这五步打通，MVP 就成立。后续增强应建立在这条主链路稳定的前提上。
